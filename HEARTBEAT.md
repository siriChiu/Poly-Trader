# HEARTBEAT.md — Poly-Trader 心跳任務（5 分鐘循環）

## 🔄 心跳完整流程

每次心跳必須嚴格執行以下 **Step 0 ~ Step 6**，不可跳過。

---

### Step 0: 閱讀 AI_AGENT_ROLE.md（每次必讀）
- **每次心跳必須讀取 `Poly-Trader/AI_AGENT_ROLE.md`**，完整閱讀
- 了解當前角色、紀律、邊界、P0 問題清單
- 確認系統架構、目標（準確度 > 90%）
- 不可跳過此步驟，即使是例行心跳

### Step 1: 數據收集
- 運行 `Poly-Trader/dev_heartbeat.py` 檢查檔案結構和模組語法
- 運行 collector 收集最新數據
- 記錄最新 raw/features/labels 數量、BTC 價格、衍生品數據

### Step 2: 感官表現分析
- 計算每個感官的 IC（Information Coefficient）對 labels
- 檢查特徵品質：std, range, unique_count
- 標記異常：IC < 0.05 → 🔴，std ≈ 0 → ⚠️

### Step 3: 多重會議激盪
#### Step 3.1: 六帽會議（每輪心跳皆執行）
| 帽子 | 職責 |
|------|------|
| ⚪ 白帽 | 客觀事實 — 數據、IC、測試結果 |
| 🔴 紅帽 | 直覺感受 — 系統給人什麼感覺 |
| ⚫ 黑帽 | 風險缺陷 — 什麼可能出錯 |
| 🟡 黃帽 | 正面價值 — 什麼在正常工作 |
| 🟢 綠帽 | 創意方案 — 有什麼新方法 |
| 🔵 藍帽 | 管控決策 — 下一步、優先級 |

#### Step 3.2: 迪士尼會議（有可行方案時執行）
| 角色 | 職責 |
|------|------|
| 🎵 夢想家 | 理想的會是什麼樣子？ |
| 🚫 批評家 | 方案有什麼漏洞？ |
| 🔧 現實家 | 實際上怎麼做？ |
| 🎯 裁判 | 最終決策：做不做？優先級？ |

#### Step 3.3: ORID 循環（會議→行動）
| 階段 | 內容 |
|------|------|
| **O** 客觀事實 | 收集到的數據、測試結果、IC 值 |
| **R** 感受直覺 | 對問題的感覺、潛在風險 |
| **I** 意義洞察 | 根本原因、系統性問題 |
| **D** 決策行動 | 列出 P0-P5 行動項目，記錄在 ISSUES.md / PRD.md / ROADMAP.md |

### Step 4: ISSUES 狀態
- 列出當前所有未解決問題及優先級
- 評估影響範圍和緊急程度

### Step 5: 修正與測試
#### Step 5.1: ISSUES 修正（完全自動執行，不問用戶意見）
- 高優先級（P0）問題直接修代碼
- 修改前先審閱（git diff）

#### Step 5.2: 功能測試（如果有修改代碼）
- 運行 `comprehensive_test.py` 確認不破壞現有功能

### Step 6: 回報摘要
輸出格式：
```
📊 心跳摘要 [時間]
- Raw: ?? / Features: ?? / Labels: ??
- BTC: $???
- 感官 IC: Eye=? / Ear=? / Nose=? / Tongue=? / Body=?
- 本輪修改：[做了什麼]
- 測試結果：[pass/fail]
```

## ⚠️ 重要原則
1. **每次心跳必須讀取 AI_AGENT_ROLE.md（Step 0），不可跳過**
2. **每次心跳執行 Step 0-6 全流程**
3. **D（決策）必須轉化為具體行動**
4. **不問用戶意見，發現問題直接修復**
5. **修改代碼後必須 commit**
